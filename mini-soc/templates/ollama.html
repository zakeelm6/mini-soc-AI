Qu'est-ce qu'Ollama ?

Ollama est un runtime LLM local qui héberge des modèles de langage directement sur la machine SOC, sans envoyer de données vers des serveurs externes. Le modèle utilisé est llama3 (8B paramètres, 4.7 GB).

Architecture
Modèle : meta-llama/llama-3-8b (quantifié Q4_0)
Port : :11434 — API REST compatible OpenAI
Inférence CPU : ~170-200s / incident
Semaphore : 1 analyse simultanée max (file d'attente)
Timeout : 240s avec fallback automatique
Chaque incident créé par l'Ensemble déclenche une analyse LLM asynchrone. Le LLM reçoit le contexte complet (IP, scores ML, logs représentatifs) et retourne un verdict structuré en JSON.
Format de sortie JSON
{
  "verdict": "true_positive",  // true_positive | false_positive | uncertain
  "attack_type": "ssh_brute_force",
  "confidence": 0.92,            // 0.0 – 1.0
  "summary": "192.168.122.231 a tenté...",
  "evidence": ["234 tentatives SSH", ...],
  "recommended_actions": ["Bloquer IP", ...],
  "severity_assessment": "critical",
  "ml_scores_comment": "IF=0.9 RF=0.98..."
}
Ce JSON structuré est stocké dans l'index soc-incidents et affiché dans le panneau d'analyse de chaque incident. Il constitue la source de vérité pour l'amélioration des modèles ML.
Boucle d'Amélioration ML/DL — Comment Ollama aide les modèles
1. Logs bruts
auth.log, apache, syslog — données non labellisées
→
2. Modèles ML
IF + RF + DL + Rate → scores d'anomalie par IP
→
3. Ensemble
Vote ≥2/4, score pondéré → incidents créés
→
4. Ollama LLM
Analyse contextuelle → verdict + type + confiance
→
5. Labels générés
TP/FP/uncertain + type d'attaque → dataset enrichi
→
6. Ré-entraînement
RF / DL retrained sur données LLM-labellisées
La boucle crée un système auto-supervisé : les modèles ML détectent des anomalies, Ollama les valide/invalide avec son raisonnement contextuel, et les verdicts deviennent des labels de haute qualité pour ré-entraîner les modèles. Contrairement aux labels manuels (coûteux), le LLM labellise en continu 24/7.
Impact sur Random Forest
Problème actuel RF labellise uniquement par IP connue (192.168.122.231/122.1). Toute nouvelle IP attaquante sera classée normale.
Apport LLM :
Ollama valide chaque nouvelle anomalie comme TP ou FP
Les IPs validées comme TP alimentent ATTACK_IPS dans rf_detector.py
Le dataset grandit à chaque cycle : 8245 → +N logs
F1=1.0 maintenu même sur nouvelles IPs attaquantes
Impact sur Autoencoder DL
Problème actuel L'autoencoder apprend un profil "normal" statique. Si le trafic normal change, le seuil de reconstruction devient inadapté.
Apport LLM :
Identifie quels incidents DL sont des FP (normal mal classé)
Ajuste dynamiquement le seuil DL_THRESHOLD de l'ensemble
Les logs FP ré-entraînent l'autoencoder pour élargir "la normale"
Réduit les faux positifs DL sans perdre en sensibilité
Impact sur Isolation Forest
Problème actuel IF est non-supervisé : ne sait pas si une anomalie est une vraie attaque ou un comportement inhabituel mais légitime.
Apport LLM :
Fournit une supervision post-hoc aux anomalies IF
Calibre IF_THRESHOLD en réduisant FP LLM-confirmés
Le type d'attaque LLM révèle quels features IF exploite mal
Permet d'ajouter des features ciblées (ex: failed_attempts_ratio)
Impact sur Rate Detector & Ensemble
Opportunité Le Rate Detector utilise des seuils fixes (10/30 logs/5min). Les poids de l'ensemble sont statiques (IF×0.30, RF×0.35…).
Apport LLM :
Analyse les patterns Rate TP/FP → seuils adaptatifs
Corrélation verdict LLM vs votes modèles → ajuste les poids
Si RF vote toujours avec LLM : W_RF augmente
Si Rate crée trop de FP : W_RATE diminue

Auto-étiquetage TP/FP par l'IA

Le service auto_labeler analyse automatiquement les incidents avec une analyse Ollama (llama3, confiance ≥ 75%) et applique le verdict sans intervention humaine : true_positive → statut in_progress, false_positive → statut closed. Après 3 nouveaux labels, le RF est ré-entraîné automatiquement.
